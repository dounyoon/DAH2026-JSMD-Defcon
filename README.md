## 1. 해커톤 개요 및 의미

DAH 2026(Defense AI Cyber Security Hackathon)은 LIG D&A가 주최하는 방산 AI 공방전 해커톤이다.
일반적인 CTF나 코딩 대회가 아니라, 위성망·드론(UAV/UGV) 시뮬레이션 환경에서
공격 AI 에이전트와 방어 AI 에이전트가 사람의 개입 없이 실시간으로 겨루는 대회다.

본 대회의 핵심은 채점식에 있다.

```
총점 = (공격 점수 + 방어 점수) × 가용성
```

여기서 가용성이 덧셈이 아니라 곱셈 항이라는 점이 전부다. 공격이나 방어를 아무리 잘해도
가용성이 무너지면 총점이 같이 무너진다. 방어 시스템이 공격을 막아내더라도 그 과정에서
CPU·메모리를 과소모해 드론 제어 신호가 끊기면, 실제 전장에서는 무인기 추락으로 이어진다.
그래서 이 대회는 "적을 얼마나 제압했는가"가 아니라 "싸우면서도 임무를 계속 유지할 수 있는가"를 평가한다.

우리가 만들어야 하는 것은 이 세 가지를 동시에 만족하는 자율 에이전트다.
공격은 상대의 가용성을 깎되, 방어는 위협만 정확히 걷어내고 자신의 가용성은 지켜야 한다.

## 2. 레퍼런스 및 적용 배경

### 참고한 레퍼런스

- CybORG CAGE Challenge 3 (드론 스웜(=무리) 시나리오): https://github.com/cage-challenge/cage-challenge-3
- CybORG (Cyber Operations Research Gym): https://github.com/cage-challenge/CybORG

CybORG는 TTCP(미·영·캐·호·뉴 5개국 국방 과학기술 협력체)의 CAGE Working Group이 개발·유지하는,
자율 사이버 방어 AI 연구용 공개 시뮬레이터이다. 그중 CAGE Challenge 3(이하 CC3)가 군용 드론 스웜을 대상으로 한다.

### 해당 레퍼런스 선정 배경

CC3의 시나리오는 이번 대회 도메인과 거의 그대로 겹친다. 국경 정찰용 자율 드론 스웜을 배경으로,
드론마다 방어 에이전트(Blue)가 한 개씩 분산 배치되고, 공격 에이전트(Red)가 드론을 장악하려 하며,
정상 사용자(Green)의 통신 성공률로 가용성을 측정한다. 이 구조가 대회의 "공격 vs 방어 vs 가용성"과
정확히 대응한다. 또 국방 협력체가 만든 벤치마크라 방산 도메인 적합성도 자연스럽게 확보된다.

(참고 : 최신 챌린지는 CC4(2024)이지만, CC4는 드론 스웜이 아니라 기업 네트워크·기지국 시나리오로 초점이 옮겨갔다.
드론 스웜 공방전이라는 우리 도메인에는 CC3가 더 맞아서 CC3를 선택했다.)

### 현재 프로젝트가 레퍼런스를 어떻게 가져왔나 (팀원용 설명)

솔직하게 말하면, 지금 단계는 CC3를 거의 그대로 가져와 돌아가게 만든 뒤, 그 위에 우리 에이전트의 뼈대까지 얹은 수준이다.
우리가 한 일은 세 가지다.

1. 최신 파이썬에서 돌아가게 버전을 고정했다.
   CC3의 드론 시나리오는 구형 라이브러리(gym 0.23.1)에 묶여 있어서 최신 파이썬(3.12+)이나 numpy 2.x에서 깨진다.
   그래서 Python 3.11 + numpy 1.23.5 + gym 0.23.1 조합으로 고정해 재현 가능하게 만들었다. (`requirements.txt` 참고.)

2. `main.py` 하나로 도는 실행 계층을 얹었다.
   CybORG 내부 코드는 건드리지 않고, 그 위에 환경 생성·에피소드 실행·채점을 자동으로 이어주는
   우리 코드(`src/`)만 새로 만들었다. 대회 채점식을 계산하는 `scoring.py`도 여기 포함된다.

3. 규칙 기반 공격·방어 에이전트의 1차 버전을 만들었다.
   `src/attack_agent.py`, `src/defense_agent.py`에 들어 있으며, `config.py`의 `USE_CUSTOM_AGENTS=True`로 켜면 동작한다.
   (끄면 CybORG 딸림 샘플 에이전트가 대신 돈다.)

즉 지금은 규칙 기반 뼈대까지 완성된 단계다. 여기에 자원 최적화·신규 공격 시나리오·강화학습을 얹는 것이 앞으로 할 일이다(7번 절 참고).

## 2-1. 중간 정리

우선 여기서 헷갈리기 쉬운 부분이라 짚고 간다. 이 프로젝트는 세 덩어리로 나뉜다.

**① 환경 (CybORG)** — 드론 스웜 전장을 돌려주는 시뮬레이터, 즉 게임판이다. 여기에는 바로 테스트해보라고 끼워둔
연습용 샘플 플레이어(무작위 방어 RandomAgent, 기본 공격 웜)도 딸려 있다. 그래서 "CybORG는 환경"이라는 말과
"CybORG에 기본 에이전트가 들어 있다"는 말이 둘 다 맞다. 게임기에 연습용 NPC가 딸려온 셈이다.
용량이 커서 이 저장소에는 포함하지 않고, 각자 내려받는다.
(단, config.py를 확인해보면, USE_CUSTOM_AGENTS = False라고 표시되어 있을 것이다. 이는 방금과 같이 제시한 CybORG에서 제시한
샘플 플레이어이고, 이를 True로 바꾸어야 우리의 에이전트가 실행된다. 우리 에이전트는 아래와 같다.)

**② 우리 실행 계층 (`main.py`, `src/`)** — CybORG를 켜고, 공방전을 돌리고, 대회 채점식으로 점수를 매기는 조종 코드다.
CybORG 내부는 건드리지 않고 그 위에 얹은 우리 코드다.

**③ 우리 에이전트 (`src/attack_agent.py`, `src/defense_agent.py`)** — 게임판에 우리가 직접 내보내는 선수다.
`config.py`의 `USE_CUSTOM_AGENTS` 값이 지금 누가 뛰는지를 결정한다.

- `False` : CybORG 환경에 같이 딸려있는 연습용 샘플끼리 싸운다. 방어가 무작위라 약하고, 가용성이 낮게 나온다.
- `True` : 우리가 만든 규칙 기반 에이전트가 대신 싸운다.

파일별 정리는 다음과 같다.

| 파일 | 하는 일 |
|------|---------|
| `main.py` | 실행 진입점. 이것만 돌리면 환경 준비 → 공방전 → 점수까지 자동 진행 |
| `config.py` | 설정값 모음 (드론 수, 스텝 수, CybORG 경로, 어떤 에이전트를 쓸지) |
| `src/environment.py` | CybORG 드론 환경을 불러와 세팅 |
| `src/runner.py` | 매 스텝 공방전을 돌리는 루프 |
| `src/scoring.py` | 대회 채점식 `(공격+방어)×가용성` 계산 |
| `src/attack_agent.py` | 우리가 만든 공격 에이전트 (규칙 기반) |
| `src/defense_agent.py` | 우리가 만든 방어 에이전트 (규칙 기반) |
| `src/reporter.py` | 결과를 화면과 `results.txt`에 출력 |
| `run.bat` | Windows 더블클릭 실행기 |
| `requirements.txt` | 필요한 라이브러리 버전 목록 |

한 줄로: 이 저장소는 CybORG(환경) 본체가 아니라, 그것을 구동·채점하고 우리 선수를 담은 우리 코드 계층이다.

## 3. 실행 방법 (Getting Started)

### 사전 준비: CybORG 내려받기

CybORG 본체는 용량이 커서 이 저장소에 포함하지 않았다. 각자 내려받아야 한다.

```bash
git clone https://github.com/cage-challenge/cage-challenge-3.git
cd cage-challenge-3
git submodule update --init
```

내려받은 폴더 안에서 `CybORG` 패키지(그 안에 `__init__.py`가 있는 폴더)의 위치를 확인하고,
`config.py`의 `CYBORG_PATH`에 그 상위 폴더 경로를 적는다. (설치를 하지 않아도 이 경로만 맞으면 실행된다.)

### Windows

```powershell
cd "C:\경로\DAH 2026"

py -3.11 -m venv .venv
.\.venv\Scripts\Activate.ps1

pip install -r requirements.txt

# config.py 의 CYBORG_PATH 를 본인 CybORG 폴더 경로로 수정한 뒤 실행
python main.py
```

이후부터는 `run.bat`을 더블클릭하면 된다.

### macOS

```bash
cd "/경로/DAH 2026"

python3.11 -m venv .venv
source .venv/bin/activate

pip install -r requirements.txt

# config.py 의 CYBORG_PATH 를 본인 CybORG 폴더 경로로 수정한 뒤 실행
python main.py
```

### 실행 파일 역할

- `main.py` : 단일 진입점. 이것만 실행하면 환경 생성 → 공방전 → 채점까지 자동으로 이어진다.
- `run.bat` : Windows 전용 더블클릭 실행기. 내부적으로 `.venv`의 파이썬으로 `main.py`를 돌린다. 설치를 대신하지는 않는다.
- `config.py` : 모든 설정(드론 수, 스텝 수, 에피소드 수, CybORG 경로, 에이전트 선택)을 모아둔 파일.
- `src/` : 환경 생성, 공격/방어 에이전트, 채점, 실행 루프, 출력.

실행 순서: 최초 1회 설치(venv 생성 → requirements 설치 → CYBORG_PATH 지정) 후, 그다음부터는 `python main.py` 또는 `run.bat` 반복.

## 4. 설정 및 출력 예시

### 설정 바꾸기

`config.py`를 열어 값만 바꾸면 된다.

| 항목 | 의미 | 예시값 |
|------|------|--------|
| `CYBORG_PATH` | CybORG 폴더 경로 | (각자 지정) |
| `NUM_DRONES` | 드론 스웜 규모 | 18 |
| `MAX_STEPS` | 에피소드당 스텝 수 | 150 |
| `NUM_EPISODES` | 반복 실행 횟수 | 3 |
| `USE_CUSTOM_AGENTS` | False=CybORG 샘플 / True=우리 규칙 기반 | True |
| `SLA_THRESHOLD` | 가용성 SLA 임계치(%) | 80 |

빠르게 확인만 하려면 `NUM_DRONES=5`, `MAX_STEPS=20`, `NUM_EPISODES=1`로 줄이면 몇 초 만에 끝난다.

### 출력 예시

아래는 `USE_CUSTOM_AGENTS=False`(CybORG 샘플)로 돌렸을 때의 예시다. `True`로 켜면 상단의 진영 표시가
"우리 공격 / 우리 방어"로 바뀐다.

```
====================================================
 DAH 2026 · 드론 스웜 AI 공방전 시뮬레이션
====================================================
 드론 5대 | 스텝 20 | 에피소드 1회
 적(Red) = CybORG 내장 웜 / 아군(Blue) = CybORG 내장 RandomAgent (raw)
----------------------------------------------------

[에피소드 1/1]
    step   0 | 생존 블루  5/5 | 누적 통신실패 3
    step  10 | 생존 블루  3/5 | 누적 통신실패 53
  공격 점수 (attack)      : 30.0  (장악 드론 3대)
  방어 점수 (defense)     : 20.0  (방어 드론 2대)
  가용성 (availability)   : 2.0%  [미달 (SLA 80%)]
  --------------------------------------------
  총점 = (공격+방어) x 가용성 = 1.0
```

결과는 화면에 표시되는 동시에 `results.txt` 파일에도 저장된다.
(CybORG가 내부적으로 쓰는 pygame이 콘솔 출력을 가로채는 경우가 있어, 파일 기록을 함께 남긴다.)

샘플(무작위) 방어는 가용성이 낮게 나오는 것이 정상이다. `USE_CUSTOM_AGENTS=True`로 우리 규칙 기반 방어를 켜면
같은 조건에서 방어 점수·가용성이 개선되는지 비교해볼 수 있다.

## 6. 필수 방산 도메인 지식

이번 해커톤에서 실제로 필요한 지식만 추려서 설명한다.

### 무엇이 일반 보안과 다른가

일반 IT 보안은 데이터를 지키는 것이 목표지만, 방산 무인기 보안은 드론이 계속 임무를 수행하게 하는 것이 목표다.
서버가 다운되면 데이터 손실이지만, 드론 제어가 끊기면 물리적으로 추락한다. 대회가 가용성을 곱셈 항으로 둔 이유가 이것이다.

### 핵심 용어

- MAVLink : 드론과 지상관제소가 주고받는 표준 통신 언어("고도 100m로 상승" 같은 명령이 이 형식으로 오간다).
  원래 인증·암호화가 없어서, 비보안 채널에서는 가짜 명령을 주입해 드론을 통째로 빼앗을 수 있다.
- 스푸핑(spoofing) : 가짜 신호나 가짜 신원으로 속이는 공격. GPS 스푸핑은 가짜 위성신호로 드론의 위치를 속인다.
- 재밍(jamming) : 강한 전파로 통신을 방해하는 것. 시뮬레이션에서는 통신 링크의 패킷 손실·지연을 강제로 악화시키는 것으로 모사한다.
- 스웜(swarm) : 여러 드론이 무리를 지어 협력·편대 유지하는 것. 한 대가 뚫리면 옆으로 번지거나(측면 이동) 편대가 무너질 수 있다.
- 가용성(availability) : 정상 서비스(드론 제어·영상 통신)가 계속 유지되는 정도. 대회에서는 정상 사용자의 통신 성공률로 측정한다.

### 공격 파이프라인: 관찰 → 분석 → 타격 → 피드백

트래픽·지연을 관찰해 취약점을 찾고, 침투에 성공하면 제어권을 탈취하며, 차단당하면 패턴을 바꿔 다시 시도해
상대의 가용성을 지속적으로 갉아먹는다.

### 방어 파이프라인: 탐지 → 격리 → 복구 → 가용성 유지

정상 트래픽의 기준선을 학습해 이상을 탐지하고, 전체망을 끊지 않고 위협 세션만 격리하며, 뺏긴 드론은 복구하되,
자원 소모를 감시해 가용성을 일정 수준(예: 80%) 이상으로 유지한다. "전체망 차단"은 정상 통신까지 죽여
내 가용성 점수를 스스로 깎으므로 금물이다.

### CybORG 액션과의 대응

| 진영 | 액션 | 의미 |
|------|------|------|
| 공격 | ExploitDroneVulnerability | 취약점 침투 |
| 공격 | SeizeControl | 제어권 탈취(드론 장악) |
| 공격 | FloodBandwidth | 대역폭 소모(재밍/DoS 모사) |
| 방어 | RemoveOtherSessions | 침입 세션만 격리 |
| 방어 | RetakeControl | 뺏긴 드론 복구 |
| 방어 | BlockTraffic / AllowTraffic | 트래픽 차단/허용(동적 페일오버) |

## 7. 공격 시나리오 및 방어 아키텍처 개선 방안

### 일단 정리 : 지금 적용된 것 vs 아직 안 된 것 (한눈에)

아래 개선 방안들 중, 규칙 기반 뼈대만 이미 코드에 들어가 있고 나머지는 앞으로 할 일이다.

| 개선 방안 | 적용 여부 | 어디에 |
|-----------|-----------|--------|
| 규칙 기반 방어 (탐지→격리→복구) | 적용됨 | `src/defense_agent.py` (`USE_CUSTOM_AGENTS=True`로 동작) |
| 규칙 기반 공격 (침투→장악→재밍) | 적용됨 | `src/attack_agent.py` |
| 대회 채점식 `(공격+방어)×가용성` | 적용됨 | `src/scoring.py` |
| 자원 예산 개념 (액션별 가용성 비용 고려) | 미적용 | 앞으로 추가 |
| 이상탐지 정교화 (정상 기준선 학습) | 미적용 | 앞으로 추가 |
| 강화학습(RL) | 미적용 | 앞으로 추가 |
| GPS 스푸핑 / MAVLink 주입 / 다중벡터 | 미적용 | 앞으로 추가 (CybORG에 없음) |
| 채점 모델 확장 (자원 소모 축) | 미적용 | 앞으로 추가 |

정리하면, 논의한 방안 중 "규칙 기반 공격·방어 + 대회 채점식"은 이미 동작하고,
실제로 점수를 끌어올릴 고급 방안(자원 최적화·신규 공격·학습)은 아직 코드에 없는 계획 단계다.

### 방어 아키텍처 개선

- 규칙 기반 방어 (적용됨). 관측을 보고 판단한다. 침입 세션이 보이면 RemoveOtherSessions로 그 세션만 격리,
  뺏긴 드론이 보이면 RetakeControl로 복구, 위협이 없으면 자원을 아낀다(가용성 보존). `defense_agent.py`에 구현되어 있다.
- 자원 예산 개념 (미적용). RetakeControl은 대역폭을 소모하고, BlockTraffic을 남발하면 정상 통신까지 막혀 가용성이 떨어진다.
  그래서 방어 액션마다 "이걸 쓰면 가용성이 얼마나 깎이나"를 따져, 임계치 아래로 가지 않는 선에서만 대응하도록 확장한다.
- 이상탐지 정교화 (미적용). 지금은 "낯선 세션이면 침입"이라는 단순 판단이다. 정상 트래픽의 기준선을 학습해 이상치를 잡는 방향으로 발전시킨다.
- 강화학습(RL)으로 발전 (미적용). 규칙 기반이 남긴 로그를 데이터 삼아, 상황별 최적 대응을 학습시킨다.
  CybORG가 심층 강화학습을 지원하므로 자연스러운 다음 단계다.

근거: CybORG/CAGE 환경은 자율 사이버 방어 에이전트를 강화학습으로 학습·평가하도록 설계된 벤치마크다.
탐지·격리·복구를 자동화하고 가용성을 지키는 방향이 이 분야의 표준 흐름이다.
참고: https://arxiv.org/pdf/2309.07388

### 공격 시나리오 개선

CC3에 이미 있는 것(취약점 침투·장악·대역폭 소모)은 우리 공격 에이전트가 이미 쓰고 있다(적용됨).
아래는 CC3에 없어서 우리가 추가하면 도메인 이해도·창의성을 어필할 수 있는 것들이다(미적용).

- GPS/GNSS 스푸핑(편대 와해). 가짜 위치 신호로 드론들이 서로 위치를 착각하게 만들어 편대를 무너뜨린다.
  CC3에 신호 계층 공격이 없어, 신규 시나리오로 제안하면 방산 특수성을 강하게 드러낸다.
  근거: https://insidegnss.com/breaking-the-formation-the-impact-of-gnss-spoofing-on-uav-swarms/
- MAVLink 명령 주입. 인증 없는 제어 프로토콜의 약점을 이용해 가짜 명령을 넣는다.
  근거: https://www.researchgate.net/publication/326860779_Empirical_Analysis_of_MAVLink_Protocol_Vulnerability_for_Unmanned_Aerial_Vehicles
- 다중벡터 공격. 재밍 + 명령 주입 + 장악을 동시에 걸어 방어의 자원을 분산시키고 그 틈에 가용성을 무너뜨린다.
  근거: https://dl.acm.org/doi/10.1145/3703625

### 채점 모델 보완 (미적용)

CC3 원본은 방어 중심이라 공격 점수 함수가 비어 있다. 우리는 `scoring.py`에서 공격 점수를 이미 정의해 두었지만(적용됨),
대회의 가용성은 통신 성공률뿐 아니라 자원 소모(SLA)까지 보므로, 가용성을 "통신 성공률 + 자원 여유" 두 축으로
확장하는 것이 규칙에 더 맞는다. 이 자원 축 확장은 아직 미적용이다.
